# 資金費率空單策略（Funding Rate Short）

入口腳本：`main_funcing_watch.py`

每小時在資金費率結算前數秒，對費率最負的幣種做空，賺取費率結算收益。

---

## 策略邏輯

Binance 永續合約的資金費率每小時結算一次（每小時 00:00）。當費率為負值時，做空方可收取費用。本策略的核心流程：

1. **HH:59:00** — 掃描所有幣對的 Premium Index，篩選「即將結算且費率低於門檻」的幣種，取前 N 名（最多 `MAX_POSITIONS`）
2. **HH:59:58** — 對每個入選幣種抓取最新市價、計算倉位大小
3. **HH:00:00 + DELAY_ORDER_MARKET_SEC** — 透過 WebSocket API 送出 MARKET 空單
4. **成交後 DELAY_ORDER_SLTP_SEC 秒** — 掛止損 / 止盈保護單

---

## 可調參數

| 參數 | 預設值 | CLI 旗標 | 說明 |
|------|--------|----------|------|
| `THRESHOLD_PCT` | `-1.0` | `--threshold` | 費率篩選門檻（%），費率必須低於此值才入選 |
| `CANCEL_THRESHOLD_PCT` | `-0.3` | `--cancel-threshold` | 若 00:00 時費率已回升高於此值，放棄下單 |
| `TP_RATE_FACTOR` | `1.5` | `--tp-factor` | TP 幅度 = `abs(費率) × factor` |
| `SL_PCT` | `0.3` | `--sl-pct` | 固定止損幅度（%） |
| `DELAY_ORDER_MARKET_SEC` | `0.1` | `--delay-market` | 等幾秒後才下市價單（避免在結算瞬間被收費） |
| `DELAY_ORDER_SLTP_SEC` | `80` | `--delay-sltp` | 成交後等幾秒再掛 SL/TP（讓 Binance 倉位帳本更新） |
| `LEVERAGE` | `5` | `--leverage` | 槓桿倍數 |
| `POSITION_RATIO` | `0.15` | `--position-ratio` | 每筆動用可用餘額的比例（15%） |
| `MAX_POSITIONS` | `2` | `--max-positions` | 最多同時下單幾個幣種 |
| `USE_TESTNET` | `False` | `--testnet` | 是否使用 Binance 測試網 |

## 執行時序

```
HH:59:00  fetch_premium_index_all()
           get_top3_symbols_nearest_funding(threshold_pct)
           → 篩選出費率 < THRESHOLD_PCT 且即將結算的幣種
           → 印出入選清單：symbol / fundingRatePct / nextFundingTime

HH:59:58  （每個入選幣種各開一個 async task，並行執行）
           fetch_all_latest_prices_v2() → price_5958
           get_account()               → available_balance
           calc_order_qty()            → qty

HH:00:00  asyncio.sleep(DELAY_ORDER_MARKET_SEC)   ← 0.1s，讓費率結算完
           [WS_RATE_CHECK] 確認費率仍 < CANCEL_THRESHOLD_PCT
           ws_api.new_order(SELL MARKET, RESULT)   ← WebSocket API 下單
           → 收到成交回報：orderId / avgPrice / executedQty

           notify_funding_short()      ← Telegram 通知

           asyncio.sleep(DELAY_ORDER_SLTP_SEC)     ← 80s，等倉位帳本更新

           place_sl_tp_orders()        ← 掛止損 / 止盈
```

多個幣種透過 `asyncio.gather` 並行執行，互不阻塞。

## 倉位計算

### 下單數量

```
notional = available_balance × POSITION_RATIO × LEVERAGE
qty      = floor(notional / price_5958, qty_precision)
```

進場參考價固定使用 **59:58 的快照價**，避免 00:00 再打一次 REST 增加延遲。

### 止盈 / 止損

以成交均價 `avg_price` 為基準：

| 方向 | 計算式 | 邏輯 |
|------|--------|------|
| 止盈（TP） | `avg_price × (1 − abs(rate) × TP_RATE_FACTOR / 100)` | 空單往下走才獲利 |
| 止損（SL） | `avg_price × (1 + SL_PCT / 100)` | 空單往上反彈 SL_PCT% 即止損 |

以 `rate = -1.0%`、`TP_RATE_FACTOR = 1.5`、`SL_PCT = 0.3`、進場 `avg_price = 100` 為例：
- TP = `100 × (1 − 0.015) = 98.50`（下跌 1.5%）
- SL = `100 × (1 + 0.003) = 100.30`（上漲 0.3%）

---

## SL/TP 掛單細節

- **SL** → Binance algo STOP_MARKET（CONTRACT_PRICE 觸發，close_position=true）
- **TP** → 限價單 reduce_only GTC（掛在委託簿，收 maker 手續費）

### -2021 異常處理

若掛止損時收到 `-2021 Order would immediately trigger`（觸發價已被穿越），自動判斷方向：

| 情況 | 現價位置 | 處理 |
|------|----------|------|
| **止損被穿越**（虧損） | 現價 ≥ SL 價 | 直接市價平倉，停損 |
| **止盈被穿越**（超額獲利） | 現價 ≤ TP 價 | 把止損掛在原 TP 價（鎖住最低利潤），止盈目標延伸 `trail_extension_pct`（預設 0.3%）繼續追 |
| 保護性止損也遇到 -2021 | 行情急速反彈 | 直接市價平倉 |

## 費率確認機制

HH:59:00 掃描到的費率會做為初始值；透過 WebSocket mark price stream 訂閱持續更新到 00:00。若 WS stream 在 Taiwan 環境無法收到資料（fstream.binance.com 有地區限制），則維持 59:00 的快照值作為 fallback。

`CANCEL_THRESHOLD_PCT`（預設 -0.3%）：若 00:00 時費率已回升高於此值，表示結算收益不足，放棄下單。

---

## Skip 條件

| 時間點 | Skip 原因 |
|--------|-----------|
| HH:59:00 | 無符合費率門檻的幣種 |
| HH:59:58 | 取價失敗 |
| HH:59:58 | `qty <= 0`（餘額不足） |
| HH:00:00 | 費率回升超過 `CANCEL_THRESHOLD_PCT` |

## 啟動方式

```bash
# 預設參數
python main_funcing_watch.py

# 自訂參數範例
python main_funcing_watch.py \
    --threshold -0.5 \
    --tp-factor 2.0 \
    --sl-pct 0.5 \
    --delay-sltp 60 \
    --max-positions 1
```

程式會持續執行，每小時在 HH:59:00 喚醒一次。
